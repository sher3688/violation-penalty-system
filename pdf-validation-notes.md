# 月度罰單 PDF 驗證紀錄

本次以 `/home/ubuntu/monthly-penalty-export-verification.pdf` 作為實際輸出檔驗證樣本。從 `pdfinfo` 可確認輸出為 **A4 尺寸**，共 **4 頁**；`pdffonts` 顯示已嵌入 `NotoSansCJKtc-Regular`，代表繁體中文字型嵌入成功。`pdftotext` 亦可抽取出通知單號、戶號、違規日期、違規類型、地點、金額與狀態等欄位文字，說明內容並未遺失。

不過，PDF 視覺檢視結果顯示，目前頁面雖然有頁首、摘要區塊、表格框線與頁尾，但明細列文字在畫面中幾乎不可見，造成表格看似空白。這表示目前的問題不在資料缺漏，而在 **PDF 版面排版或文字繪製方式**，需要進一步修正月報表格列的樣式或欄位結構，之後再重新產生實際 PDF 驗證檔。


進一步改用 `DroidSansFallbackFull.ttf` 後，`pdffonts` 已確認 PDF 內同時嵌入 `DroidSansFallback` 字型，且仍維持 A4 共 4 頁。但重新檢視實際 PDF 頁面後，表格框線與頁首頁尾都正常，**明細文字依然未顯示在頁面中**。這表示問題不單純是 OTF 與 TTF 相容性，而更可能與 React PDF 的表格列布局、欄寬定義、或文字元件在列內的繪製方式有關，需要改寫月報 PDF 的列排版策略後再做一次實際檔案驗證。


移除固定列高與裁切後，實際 PDF 已可抽取並視覺看見通知單號、戶號、違規日期、違規類型、地點、金額與狀態，確認明細文字遺失問題已排除。然而，首兩頁視覺檢查顯示單列高度過大、31 筆資料被分成 9 頁，且違規類型／地點與其餘欄位的垂直對齊尚不理想。後續修正將改以固定點數欄寬（預先扣除各欄內距）與每欄的 View + Text 結構，避免以百分比欄寬加內距造成 React PDF 的非預期自動換行。


卡片式排版已讓每筆明細的所有文字可見，但實際首兩頁顯示每張卡片高度被不合理拉長，首張僅容納 1 筆、後續頁僅容納 3 筆。從 `recordDetail` 與 `recordLocation` 被拉開的垂直距離判斷，主因更可能是目前 `DroidSansFallbackFull.ttf` 在 React PDF 中的字型度量（ascent/descent）異常，而非卡片 CSS 邏輯。後續需改用可被 React PDF 正確嵌入且字型度量正常的靜態繁體中文字型，再重跑實際 PDF 驗證。


最新實際檔案驗證已確認：改用 Noto Sans CJK TC 並將 `lineHeight` 改為倍率後，31 筆 2026-07 測試資料可穩定輸出為 **5 頁 A4 PDF**。第 1 頁包含摘要區與 6 筆案件；第 2 至第 4 頁各 7 筆；第 5 頁 4 筆。五頁皆可視覺辨識通知單號、戶號、違規日期、違規類型、地點、金額、狀態與連續頁碼，證明目前的直接下載 PDF 流程已具備可交付的穩定性。
